# validgo-gen — Full Design & Usage Document

> OpenAPI 3.0 → Go code generator with two-layer validation
> Repository: `github.com/sintoniastrategy/validgo-gen`

## Overview

validgo-gen reads OpenAPI 3.0 YAML specifications and produces **two Go files per spec**:

| File | Package | Contents |
|------|---------|----------|
| `<name>models/models.go` | `<name>models` | Struct definitions for request bodies, response bodies, path/query/header/cookie params, composite request/response types |
| `handlers.go` | `<name>` | Handler interfaces, chi route registration, request parsing, JSON validation functions, response writing |

**What makes it different from oapi-codegen / go-swagger:**

- **Go AST-based generation** — code is built as `go/ast` nodes, then formatted via `go/format.Node()`. No templates. Output is syntactically valid and `gofmt`-formatted by construction.
- **Two-layer validation** — raw JSON validation (`json.RawMessage` checks for required/null/nested structure) runs *before* deserialization, followed by struct-level validation via `go-playground/validator/v10` tags.
- **Handler interface pattern** — each operation produces a single-method interface. You implement only the business logic; all parsing/validation/routing/response-writing is generated.

## Installation & CLI Usage

### Build

```bash
go build -o validgo-gen ./cmd/generate.go
```

### Run directly

```bash
go run ./cmd/generate.go [flags] <file1.yaml> [file2.yaml ...]
```

### Via `go:generate` (recommended)

```go
package myapi

//go:generate go run github.com/sintoniastrategy/validgo-gen/cmd/generate.go \
//   -d ./generated -p github.com/myorg/myproject/generated \
//   api.yaml definitions.yaml
```

Then:

```bash
go generate ./...
```

### Makefile targets (in the repo)

```makefile
make generate   # go generate ./...
make test       # go test ./...
make lint       # golangci-lint run
make check      # all three sequentially
```

## Input / Output Contract

### Input

One or more OpenAPI 3.0 YAML files. The generator:
- Parses each file with `kin-openapi`
- Validates the OpenAPI document structure
- Discovers external `$ref` targets and adds them to the processing queue

### Output directory structure

Given flags `-d ./generated -p github.com/myorg/project/generated` and input `api.yaml`:

```
generated/
  api/
    handlers.go              # package: api
    apimodels/
      models.go              # package: apimodels
```

Package names are derived from the YAML filename:
- Strip extension
- Lowercase
- Remove dashes/underscores
- Models sub-package = `<name>models`

Example: `a_pi.yaml` → package `api`, models package `apimodels`.

### Generated file header

Every file starts with:
```go
// Code generated by github.com/sintoniastrategy/validgo-gen; DO NOT EDIT.
```
